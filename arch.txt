Слои системы:
=============

1. API Layer (broker/server/api.py)
   - FastAPI с Swagger UI на /api/v1/docs
   - REST эндпоинты для управления топиками и подписками
   - POST /ack — явное подтверждение доставки
   - WS /ws/{subscriber_id} — push-уведомления + ACK по сокету

2. Core Layer (broker/core/)
   - MessageBroker — оркестратор: создание/удаление каналов, маршрутизация публикаций,
     ACK, метаданные, фоновые потоки (retry каждые 0.5с, cleanup каждые 60с)
   - Topic — pub/sub: fan-out всем подписчикам, in_flight dict, replay по offset
   - Subscriber — абстракция получателя: WS (через asyncio event loop из потока)
     или callback (для embedded тестов)

3. Persistence Layer (broker/persistence/storage.py)
   - JsonlLogStorage: каждое сообщение — одна строка JSON с fsync
   - Структура на диске:
       data/
         meta.json                     ← реестр (восстановление после перезапуска)
         topics/<name>/messages.jsonl  ← append-only лог
         offsets/<sub>/<type>__<name>.json  ← позиция чтения подписчика
         dedup/producer.json           ← LRU кэш {client_msg_id: server_msg_id}

4. SDK Layer (client/)
   - BrokerPublisher:
     * генерирует client_message_id (UUID) если не передан
     * POST с timeout=5s; при ошибке — экспоненциальный retry (0.5 → 1 → 2с)
     * возвращает (message_id, offset)
   - BrokerSubscriber:
     * WebSocket подключение к /api/v1/ws/{subscriber_id}
     * при connect — отправляет subscribe-фреймы, брокер делает replay
     * после callback — отправляет ACK (или не ACKит при ошибке → retry брокера)
     * seen_ids set — дедупликация повторов


Потоки данных:
==============

Публикация (at-least-once, producer side):
  Publisher SDK
    → POST /topics/{name}/publish {content, client_message_id}
    → ProducerDedupCache.check(client_message_id)
        если найден → вернуть cached {message_id, offset}  (dedup)
        если новый  → Message.create() → Topic.publish_message()
                        → JsonlLogStorage.append() [fsync]  → offset
                        → _deliver_to_all(): в in_flight + Subscriber.deliver_message()
                        → return {message_id, offset}
    ← SDK: если timeout/5xx → retry с тем же client_message_id (broker dedup)

Доставка (at-least-once, consumer side):
  Broker
    → Subscriber.deliver_message(msg)
        → asyncio.run_coroutine_threadsafe(ws.send_json(msg), loop)
    Subscriber SDK
       ← WS frame {id, offset, content, topic, ...}
        → callback(msg)
        → WS send {"action": "ack", "message_id": msg_id, ...}
    Broker
        ← ACK → topic.ack(sub_id, msg_id)
                   → remove from in_flight
                   → save offset to disk
  Retry loop (каждые 0.5с):
    для каждого in_flight[(sub_id, msg_id)]:
      если now - sent_at > 3с:
        retry_count < max_retries → re-deliver (та же запись in_flight)
        retry_count == max_retries → DLQ, remove from in_flight

Восстановление после перезапуска:
  Broker.__init__
    → JsonlLogStorage.load_meta()           ← список топиков/подписок
    → Topic.load_from_storage()             ← читает messages.jsonl в in-memory deque
    → restore subscribers from meta
    → load_dedup_cache()                    ← восстановить LRU dedup
  При переподключении подписчика (WS):
    → {"action": "subscribe", ...}
    → broker.create_subscription() → topic.add_subscriber(replay=True)
    → load_offset(sub_id) → replay messages[saved_offset+1:]


Гарантии:
=========
- At-least-once delivery: сообщение остаётся в in_flight до получения ACK.
  Если ACK не пришёл за 3с — retry с тем же message_id.
- Producer idempotency: client_message_id дедуплицируется на брокере
  (LRU OrderedDict, 10 000 записей, персистентен на диске).
- Consumer idempotency: SDK хранит seen_ids (LRU, 10 000) для защиты от повторной
  обработки при broker retry.
- Durability: каждая запись в JSONL сопровождается flush+fsync.
- Offset recovery: при рестарте потребитель получает только новые сообщения
  (сохранённый offset читается из data/offsets/).
